API Gateways

by Priyal Walpita

API网关是一种许多API开发者用来封装API端点的典型的模式。但是，更常见的是用API网关来反设计模式。本文聚焦于API网关模式的正确使用。

一个API（应用程序接口）网关是一个位于其他后端（微）服务的接口。API网关可以用来确保安全性、可扩展性和高可用性等系统架构特性。

我们是否能不使用网关来实现我们的服务（例如RESTFull或者SOAP）？当然可以。但是单独的服务需要不少非业务具体功能，例如路由、认证、授权、TLS、日志、追踪等等。更重要的是，如果不使用网关来暴露API端点，客户端将获取后端服务的分布和部署细节。当你使用API端点集成内部前端应用时，这没什么大问题了。对于企业级API网关，它们可能被暴露给未知的需求客户端，这就不是一个好的实践。

反向代理或者负载均衡能被用作网关吗？

如果你使用网关只为了抽象一个单点后端或者结束TLS通信，反向代理或者负载均衡就足够了。但是固然，一个合适的API网关能够满足以下的功能：

  • 多类型后端（微服务）

  • 服务发现

  • 断路限制

  • 认证授权

  • 限速

  • 记录追踪

  • 路由

  • 重试逻辑

    等等..

单一网关是好是坏?

如果你部署单一网关，即使很容易管理它，你也会发现以下麻烦：

单点故障: 如果API网关不在合理地提供功能，你的整个后端通信就会失效。一个解决方案是，你可以拥有一个可扩展的API网关集群基于一个硬件负载均衡，在这负载均衡会注意到反压。

多样的通信需求会逆转一个单点API网关栈: 依赖于多样的业务功能，API需求在性能、安全和合规等方面上都是不同的。如果你有API网关集群，所有的交通会从中穿过，你不能够负载这些不同的需求。尤其是在你建设一个企业级的API网关的时候这不是一个好主意。

使DevOps功能变得困难: 如果你在后端服务执行路由改变或者协议变化，这也会影响API网关。所以，一旦你部署了一个变化，API网关需要更新来使用单一的API网关集群。

因此，最好的选择是拥有多样的API网关，具体来说，是对实际需求设计载体。

API 网关能增强两个方面
  • 企业API网关
  • 微服务API网关

企业级API网关主要用于应用市场，在市场你期望任何第三方付费后来使用你的API。大部分API集成载体都是这样，以下情形是开发企业级API网关需要注意的地方：

  • 一个分离的企业集成团队可能会实现部署.QA →Staging → UAT → Production 手工过程会被使用.
  • DevOps管道很难在这种场景里被利用
  • 客户端的错误封装（定制错误）
  • 需要监控调用率和HTTP状态。你的账单可能依赖于此
  • 我们需要更关注安全性，数据合规和标准等等

在微服务方面, 对网关的关注点如下：

  • 在生态系统内部使用，API网关只对内部客户端可见
  • CI/CD管道的高使用率以及部署类型例如Canary, Shadow, Blue-Green等等
  • API 可能暴露错误和追踪栈的所有细节
  • 非功能性的需求例如延迟和性能将会被考虑在企业水平的限制（标准、合规和安全等等）之上

API网关反模式

以下情况被视为API网关反模式：

  • 通过单一服务来路由最多的通信会烧毁网关
  • 没有考虑到API网关依赖的服务可扩展性和韧性因素( 例如：认证，服务发现，路由等等 )，这些服务的失效会直接影响API网关的可用性
  • 避免使用API网关作为ESB来重设路由内部通讯，API网关应该只对服务层在外部来的通信时路由
  • 确保单点故障被减轻和管理
  • 使API网关孤立以及不被监控

以下是我们可以采用的商业版或者开源产品：

  • Kong Gateway
  • Apache APISIX
  • Tyk
  • Goku
  • WSO2
  • KrakenD
  • Zuul
